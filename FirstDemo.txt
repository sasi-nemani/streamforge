KEY ARCHITECTURAL DECISIONS
════════════════════════════

1. CONNECTOR ABSTRACTION (pluggable event sources)
   All event ingestion flows through `StreamConnector` ABC.
   Implementations: File (NDJSON), Kafka, Mock (testing).
   Adding GCP Pub/Sub, AWS Kinesis, Azure Event Hubs = one new class, zero
   changes to inference or drift logic.

2. LLM INFERENCE IS ONE-TIME ONLY
   The LLM call happens once during `streamforge init`. After that, all drift
   detection is pure statistical (chi-squared, binomial z-test, PSI).
   Cost per schema: ~$0.02. Cost per monitoring cycle: $0.00.

3. SCHEMA AS CODE (plain YAML)
   Schema stored as human-readable schema.yaml, committed to Git.
   Engineers can diff it in PRs, edit it manually, and version it.
   No proprietary binary format. No vendor lock-in.

4. DRIFT CLASSIFICATION (3-class)
   DRIFT    → breaking, trigger CI gate, page on-call
   EVOLUTION → non-breaking (new optional field), open schema PR, no alert
   NOISE    → low-confidence signal, suppress (prevents alert fatigue)

5. MULTI-SCHEMA CLUSTERING (per-event-type sub-schemas)
   Streams with mixed event types (payment, refund, chargeback) get one schema
   file with per-cluster sub-schemas. Drift is reported at cluster granularity.
   Routing regression detection (cluster disappears → producer bug flagged).

6. SEVERITY TIERING (3 tiers)
   TIER 1 → trivial, non-breaking (new optional field). Log only.
   TIER 2 → breaking but auto-correctable (type changed epoch→ISO). Alert.
   TIER 3 → critical, data integrity risk (required field removed). Block CI.

7. LAYERED CONFIGURATION
   CLI flags > env vars > topic YAML > environment YAML > global defaults.
   Per-topic config: config/topics/events.payments.yaml
   Per-environment: config/prod.yaml
   No secrets in version control.

8. WATCH LOOP STATE MACHINE
   Cold-start problem: new streams go LEARNING → STABILIZING → STABLE.
   Prevents initial churn from being misclassified as drift.
   Configurable warmup cycles, stability cycles, consecutive drift threshold.

CORE COMMANDS
══════════════

  streamforge init <stream>
    → AI-infers schema from event sample (LLM-powered, once)
    → Produces schema.yaml, inference_report.md, sample archive
    → Flags PII fields automatically
    → Supports Kafka URI (kafka://topic) and NDJSON folders

  streamforge watch <stream>
    → Continuous monitoring loop (configurable interval)
    → Detects drift with statistical tests (no LLM cost)
    → Classifies each drift: DRIFT / EVOLUTION / NOISE
    → Writes timestamped drift reports on detection
    → Posts to Slack/PagerDuty if configured
    → Exits 1 on TIER 3 drift (blocks CI pipelines)

  streamforge plan <stream>
    → One-shot drift check (like "terraform plan")
    → Returns exit code 0 (no drift) or 1 (breaking drift)
    → Designed for CI/CD integration

  streamforge profile <stream>
    → Multi-schema profiling for mixed-event-type streams
    → Discovers and names sub-clusters automatically
    → Produces profile.yaml + profile_report.md

  streamforge report <stream>
    → Print current schema + drift history summary to console
    → Shows per-field confidence, PII flags, enum values

  streamforge export <stream> --format <fmt>
    → Convert schema to: json-schema | avro | protobuf | ksqldb | flink
    → Enables downstream integration without vendor lock-in

  streamforge vcs <action>
    → Auto-commit schema changes to Git
    → Auto-open PRs on GitHub/GitLab for schema evolution

  streamforge consumers <stream>
    → List subscribed consumers + auto-correct status
    → See which downstream systems might be affected

  streamforge discover --brokers <brokers> [--schemas-dir schemas]
    → Lists ALL topics on the Kafka cluster
    → Shows which topics have schema contracts and which don't
    → Output: "Discovered 47 topics | Monitored (3) | Unmonitored (44)"
    → The opening demo moment: instant visibility into governance gaps

  streamforge kafka-ping <topic> --brokers <brokers> --timeout <seconds>
    → Connectivity check: exits 0 if Kafka reachable, 1 if not
    → Used by demo scripts and health checks before starting the watch loop

  streamforge incident-report <stream> [--since 30d] [--min-tier 2]
    → Generates a structured report of all schema incidents caught
    → Includes field names, drift types, timestamps, and tier
    → Designed to be shared with engineering managers and VPs
    → Example: "2 Tier 3 incidents prevented in the last 30 days"

DRIFT DETECTION
════════════════

  - Binomial z-test: presence rate change (field disappears/appears)
  - Chi-squared test: type distribution change (field becomes mixed)
  - Chi-squared test: enum drift (new enum values appear)
  - PSI (Population Stability Index): numeric field distribution shift
  - Conservative α=0.01 (prevents false positive alerts)
  - Type refinement awareness: string→UUID is NOT drift, it's precision gain
  - Timestamp format equivalence: epoch↔ISO is TIER 2, not TIER 3
  - Type widening rules: int→float is safe (EVOLUTION), float→int is breaking
  - Presence hysteresis: sustained changes only (no day-of-week noise)

DRIFT CLASSIFICATION
═════════════════════

  TIER 1 — Non-breaking
    - New optional field added
    - Presence rate increased
    - Type refined (string → UUID)

  TIER 2 — Breaking but auto-correctable
    - New required field
    - Type widened (int → float)
    - Timestamp format equivalent change (epoch → ISO)
    - Enum values expanded

  TIER 3 — Critical (human required)
    - Required field removed
    - Type narrowed incompatibly
    - New PII field detected
    - Presence drops >50 percentage points

  DRIFT   → existing schema violated, block CI, page engineer
  EVOLUTION → additive change, open schema PR, no human alert
  NOISE   → low-confidence signal, suppress (prevents alert fatigue)

Imagine your company is a busy airport.

Every second, hundreds of "flight status updates" flow through your systems:
  "Flight BA123 departed at 14:30, from LHR, to JFK, 245 passengers"

These updates power everything: departure boards, baggage systems, crew
scheduling, passenger apps.

Now imagine one day, without warning, someone changes the format. Instead of
"245 passengers", they start sending "245.0" (a decimal). Or they rename the
field from "passengers" to "pax_count". Or they stop sending it entirely.

Your baggage system breaks. Your crew scheduling goes wrong.
Hundreds of passengers are affected before anyone notices.

StreamForge is like having a highly trained inspector at the gate who:

  1. STUDIES what the updates normally look like (this is the AI part — it
     reads thousands of updates and writes down the "rules")

  2. WATCHES every new update that comes in and checks it against the rules

  3. RAISES AN ALARM the moment something doesn't match — instantly,
     not 6 hours later when the damage is done

  4. TELLS YOU EXACTLY WHAT CHANGED in plain English:
     "Field 'passengers' was always a whole number. It's now a decimal.
      34% of today's flights are affected. Severity: Medium."

  5. BLOCKS THE CHANGE if it's serious enough (like a safety officer who
     won't let a non-compliant aircraft onto the runway)

The magic is that it learns automatically (you don't write rules by hand),
it watches continuously (not once a day), and it knows the difference between
a harmless change (someone added a new optional field) and a dangerous one
(someone removed a field everything depends on).

THE TECHNICAL PROBLEM
══════════════════════

You have Kafka topics. They carry JSON events. Your producers are owned by
4 different teams. Your consumers (Flink jobs, Spark pipelines, ML models,
REST APIs) are owned by 6 others. There is no formal contract between them.

Every schema change is a roll of the dice:
  - Producer team adds a field → consumers that fail on unknown fields break
  - Producer team removes a field → consumers that require it break
  - Producer team changes a type → consumers that deserialize strictly break
  - Producer team renames a field → everything breaks

Confluent Schema Registry with Avro helps — if you have the discipline to
use it. Most teams don't, because writing Avro schemas by hand is tedious,
your producers are in 4 languages, and schema enforcement in dev environments
is a joke.

HOW STREAMFORGE SOLVES IT
════════════════════════════

  Step 1: INFER (once, ~$0.02 per stream)
  ───────────────────────────────────────
  Run `streamforge init kafka://events.payments`

  StreamForge samples 400 events from Kafka, sends them to an LLM with
  a structured tool-use prompt, and receives back a typed schema:
  {field, type, required, nullable, presence_rate, enum_values, confidence, pii}

  It applies post-inference schema hints (regex-based type boosting) and
  PII detection (pattern + field-name heuristics). It writes schema.yaml to
  your repo and registers it in Confluent Schema Registry.

  Cost: one LLM call. ~$0.02. Never again.

  Step 2: WATCH (continuous, $0.00 per cycle)
  ────────────────────────────────────────────
  Run `streamforge watch kafka://events.payments` (or deploy as a K8s CronJob)

  Every 30 seconds (configurable), it samples 200 events and runs:
    - Binomial z-test: did field presence rates change significantly? (α=0.01)
    - Chi-squared test: did the type distribution change?
    - PSI: did numeric field distributions shift?
    - Enum diff: are new categorical values appearing?
    - Cluster routing: did a sub-cluster disappear or a new one appear?

  All statistics. No LLM. Sub-100ms per cycle.

  Step 3: CLASSIFY + ACT (tiered response)
  ─────────────────────────────────────────
  Each detected change is classified:

  DRIFT class + TIER 3 → Slack alert, PagerDuty incident, `streamforge plan`
                          returns exit code 1, blocking your deployment pipeline

  EVOLUTION class → Silent schema PR opened in GitHub/GitLab. New optional
                    field gets added to schema.yaml. No engineer interrupted.

  NOISE class → Suppressed. Statistical signal below confidence threshold.
                Alert fatigue prevented.

13. NO IDEMPOTENCY GUARD ON INFERENCE [P2]
    Problem:  Running `streamforge init` twice on the same stream creates a
              second schema.yaml overwriting the first. If a human edited
              schema.yaml manually, their changes are lost.
    Fix:      Check if schema.yaml exists → prompt user (--force to overwrite).
              Or: merge new inference with existing human edits.

14. STATISTICAL TESTS: NO MULTIPLE COMPARISONS CORRECTION [P2]
    Problem:  When monitoring a stream with 50 fields, running 50 independent
              hypothesis tests at α=0.01 gives an expected ~0.5 false positives
              per cycle (50 × 0.01). Bonferroni or FDR correction needed.
    Fix:      Apply Benjamini-Hochberg FDR correction across all field tests
              per cycle. Only flag drifts where corrected p-value < α.
    Impact:   Reduces false positive rate from ~0.5/cycle to near zero.

15. NO INTEGRATION TEST SUITE AGAINST REAL KAFKA [P2]
    Problem:  All Kafka tests use the mock connector. If the kafka.py connector
              has a real-world bug (SASL auth, SSL, offset management), it
              won't be caught until production.
    Fix:      Add a docker-compose test profile (Kafka + tests) that runs
              integration tests against a real Kafka broker in CI.

16. OPENTELEMETRY METRICS NOT EXPORTED [P2]
    Problem:  The Kafka offset metrics loop writes to a log file. There's no
              OTLP export to Datadog, Grafana, or any observability platform.
    Fix:      Add OpenTelemetry SDK. Emit gauges:
              streamforge.drift.detected (count), streamforge.schema.confidence
              (gauge), streamforge.events.sampled (counter).

17. SCHEMA DRIFT HISTORY NOT QUERYABLE [P2]
    Problem:  Drift reports are stored as timestamped markdown files. There's
              no way to query "how many TIER 3 drifts did events.payments have
              in the last 30 days?" without reading every file.
    Fix:      Write a drift_index.json alongside each drift report. Maintain
              a rolling index for dashboard queries. ~1 day.

18. NO GRACEFUL SHUTDOWN ON SIGTERM [P2]
    Problem:  In Kubernetes, when a pod is killed, SIGTERM is sent. The watch
              loop does not handle SIGTERM gracefully — it may die mid-write,
              leaving a partial drift_report.
    Fix:      Install signal.SIGTERM handler that sets a stop flag, completes
              the current cycle, then exits cleanly.

19. MULTI-SCHEMA: UNKNOWN CLUSTER THRESHOLD TOO LOW BY DEFAULT [P2]

              alarms. events.all sets 0.30 explicitly, but other topics don't.
    Fix:      Raise global default to 0.10 in default.yaml with clear comment.

              overflows.
    Fix:      Use Streamlit's column layout with responsive widths.
              Low priority unless investors are checking on phone.
