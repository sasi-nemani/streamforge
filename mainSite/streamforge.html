Technical Details

How it works.

1. Non-Destructive Sampling

StreamForge never consumes messages from your queues. We use platform-native read-only mechanisms:

  • Kafka: Dedicated consumer group with independent offsets — your existing consumers are unaffected
  • SQS: ReceiveMessage with VisibilityTimeout=0 (peek mode) — messages remain in queue
  • IBM MQ: Browse mode — non-destructive read without removing messages
  • RabbitMQ: Passive queue declaration + basic.get with no-ack

Result: Zero impact on message delivery. Zero risk to production consumers.

2. Statistical Type Inference

Schema inference uses distribution analysis, not guesswork. For each field, we compute:

  • Presence rate: % of events containing the field (determines required vs optional)
  • Type distribution: Statistical breakdown of observed types per field
  • Value patterns: Regex matching for semantic types (UUID, email, ISO timestamp, etc.)
  • Cardinality analysis: Distinct value count to detect enums vs free-form strings

Confidence scores are computed per-field. Low-confidence inferences are flagged for human review.

3. LLM-Assisted Semantic Understanding

For ambiguous fields where statistical inference is insufficient, StreamForge uses LLM analysis to understand semantic meaning. Critical: Your actual data never reaches the LLM.

A
PII Detection (Local)

Pattern matching + heuristics identify potential PII fields locally. No external calls.

B
Pseudonymization

Before any LLM call, field values are replaced with synthetic examples that preserve structure but contain no real data.

"john.doe@acme.com" → "user_1@example.com" "4532-1234-5678-9012" → "XXXX-XXXX-XXXX-0000" "192.168.1.100" → "10.0.0.1"
C
Semantic Inference

LLM receives only: field names, pseudonymized samples, and structural metadata. It returns semantic classifications (e.g., "this looks like a correlation ID" or "this appears to be a currency amount in minor units").

D
Statistical Fallback

If LLM is unavailable or confidence is low, inference falls back to pure statistical methods. The system degrades gracefully — never blocks on external dependencies.

Data residency: LLM calls can be disabled entirely, routed to on-premise models, or restricted to specific providers. Pseudonymization is applied regardless of destination.

4. Statistical Drift Detection

Continuous monitoring uses proven statistical tests with configurable sensitivity:

  • Binomial z-test: Detects presence rate changes (field appearing/disappearing)
  • Chi-squared test: Detects type distribution shifts (string → integer mixing)
  • Population Stability Index (PSI): Detects numeric distribution drift
  • Enum diff: Detects new categorical values appearing
  • Benjamini-Hochberg FDR: Controls false positive rate at scale (critical for 100+ field schemas)

Default α=0.01 (1% false positive rate). Tunable per-stream for different sensitivity requirements.

5. PII Detection Pipeline

Multi-layer detection ensures comprehensive PII coverage:

  • Field name heuristics: "email", "phone", "ssn", "card_number" patterns
  • Regex patterns: Email format, phone formats, credit card Luhn validation, IP addresses
  • Semantic analysis: LLM identifies non-obvious PII (e.g., "loyalty_number" containing SSN format)
  • 10 PII categories: Email, phone, name, passport, card, IP, DOB, national ID, address, loyalty ID

PII detection runs locally first. LLM is only consulted for ambiguous cases, always with pseudonymized data.

6. Security Architecture

  • Read-only access: StreamForge never writes to your message brokers
  • No message storage: Events are sampled, analyzed, and discarded — not persisted
  • Pseudonymization by default: Real values never leave your infrastructure for LLM calls
  • Configurable LLM routing: Use Claude, GPT-4, on-premise models, or disable entirely
  • Audit logging: Every inference decision is logged with reasoning for compliance review
  • No network egress required: Can run fully air-gapped with statistical-only mode
