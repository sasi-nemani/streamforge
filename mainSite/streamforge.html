How it Works

One command to set up.
Continuous protection.

01
INFER — one-time setup per stream

Statistical inference learns your schema

One command. StreamForge samples hundreds of events from your Kafka topic, analyzes type distributions, presence rates, and value patterns to infer a fully typed schema — field types, required fields, enum values, confidence scores, PII flags.

It writes a human-readable schema.yaml to your repo. Readable, diffable, editable. No proprietary format. No vendor lock-in.

Terminal 
$ streamforge init kafka://events.payments

✓ Loaded 400 events from Kafka
✓ Sampled 400 events (all)
📊 Inferring schema via statistical analysis...
✓ Schema inferred — 12 fields, confidence 0.91
⚠ PII detected: user.email, user.name, card_last_four
✓ Written: schemas/events.payments/schema.yaml
✓ Registered in Confluent Schema Registry
02
WATCH — continuous monitoring

Statistical drift detection, always on

Every 30 seconds, StreamForge samples 200 events and runs a battery of statistical tests — binomial z-test for presence rate changes, chi-squared for type distribution shifts, PSI for numeric drift, enum diff for new categorical values.

Pure statistics. Sub-100ms per cycle. Lightweight and deterministic.

03
CLASSIFY & ACT — tiered response

Block bad deployments. Open PRs. Kill alert fatigue.

Not all schema changes are equal. StreamForge classifies every detected change and responds accordingly — no manual triage needed.

TIER 1 Evolution

New optional field, presence rate up. Auto-PR opened. No human interrupted.

TIER 2 Breaking

Type widened, enum expanded, timestamp format changed. Slack alert sent. Auto-correctable.

TIER 3 Critical

Required field removed, new PII detected, presence drops 50pt+. CI blocked. PagerDuty paged.
